# Issue: Point-in-Time バックテストの導入

## 概要 (Overview)
過去のバックテスト実行時点において、「その時点で本当に入手可能だったデータ（財務情報など）」のみを用いてシミュレーションを行うアーキテクチャへの改修を行う。

## 背景・課題 (Background / Problem)
- 現状のバックテストは、企業が後から修正した財務データ（遡及修正分）や、テスト時点では未発表だった情報（Look-ahead bias）を含んでしまう可能性がゼロではない。
- より厳密な戦略評価（カーブフィッティングの排除）を行うためには、Point-in-Time（特定の時点におけるスナップショット）データの取り扱いが必要不可欠。

## 修正方針 (Proposed Solution)

### 1. 過去データのスライシング（Point-in-Time）の厳密化
- 財務データ（EDINETからの有報提出日など）の「利用可能日（Available Date）」を管理・意識したデータ構造へ移行する。
- バックテストエンジン (`src/backtester.py`) が、各シミュレーション日 `t` において、`Date <= t` の情報のみをフェッチ（あるいはフィルタリング）して戦略評価に渡すように強制する。

### 2. ローリング・バックテスト (Rolling Window Backtesting) の導入
- 固定の期間（例: 2020年〜2023年）だけでなく、ウォークフォワードテスト（Walk-Forward Testing）のように、学習期間とテスト期間をローリングさせながらパフォーマンスを評価する仕組みを導入する。

## タスクリスト (Tasks)
- [ ] バックテストエンジン (`backtester.py`) での利用可能日（Available Date）ベースのフィルタリングロジックの実装
- [ ] EDINET/yfinance 取得データのタイムスタンプ付与ルールの見直し・厳密化
- [ ] Walk-Forward Testing 用のローリングバックテスト関数の作成

## 影響範囲 (Impact)
バックテストエンジン (`backtester.py`) およびデータ取得層 (`data_fetcher.py`, `edinet_client.py`)。

## 備考 (Notes)
バックテストの計算コストが増大する懸念があるため、データキャッシュ機構 (`sqlite` 等への移行) と並行して進める、あるいはその後に着手することが望ましい。
