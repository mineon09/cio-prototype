# SECデータプロンプト未反映問題の修正計画

## 問題の背景
`generate_prompt.py` を用いて米国株（XOMやAMATなど）のプロンプトを生成する際、以下の問題が発生していました。
1. プロンプトの `4. スコアカード概要` 欄で `[Qualitative 詳細] 定性分析スキップ（有報データなし）` と表示される。
2. `5. 定性データ・有価証券報告書要約` 欄が「（有報データ未取得）」となったり、長文すぎてAPI(Groq等)のトークン超過エラー(413)を引き起こして出力が崩れる。

## 原因の特定
調査の結果、以下の3点が複合して発生していることが判明しました。

1. **スコアカードへのデータ未連携**:
   `generate_prompt.py` の `collect_data_minimal()` 内で、`generate_scorecard()` を呼び出す際、API呼び出し削減（軽量版）の意図で `yuho_data` 引数に明示的に空の辞書 `{}` を渡していました。これにより、せっかく抽出したSECデータが存在してもスコアカード上では必ず「有報データなし」として扱われていました。
   
2. **トークンサイズの上限超過 (HTTP 413)**:
   `sec_client.py` は、最大8万文字の 10-K 生テキストを `raw_text` として返します。`format_yuho_for_prompt()` はこれをほぼそのまま文字列に結合するため、全体の文字数が1万文字を超えます。これが最終的なプロンプトに埋め込まれると、後続の Groq (Llama 3) などのLLM APIが `Request too large` エラーを起こし、例外処理に落ちてデータが「未取得」扱いになるケースがありました。

3. **例外時の変数上書きミス**:
   例外発生時、`yuho_summary` が適切にフォールバックメッセージ（例:「SEC取得エラー」）に更新されず、初期設定の「（有報データ未取得）」がそのまま出力される古いコードパスを通る場合がありました。

## 修正プラン

### 1. `generate_prompt.py` の修正
- `generate_scorecard` 呼び出し時に、抽出済みの `yuho_data` を**正しく渡す**ように変更します。これにより、リスクや堀の評価がスコアカード(10点満点)の算出に反映され、「定性分析スキップ」の表示が解消されます。
- `try-except` ブロック内のエラーハンドリングを見直し、SECデータの取得やプロンプト生成でエラーが起きた際に、何が原因かを明確にプロンプトに出力するようにします。

### 2. `src/analyzers.py` の修正
- `format_yuho_for_prompt()` 内で、`raw_text` をプロンプト文字列に追加する処理を見直します。
- 1万文字以上の生テキストをそのままLLMに投げると確実にパンクするため、`raw_text` の出力上限を **最大1000文字程度** に大幅に切り詰める（あるいはAIによる要約 `summary` がすでに存在する場合は `raw_text` をそもそも追記しない）ようにし、トークン消費を抑えます。

## 期待される結果
- 米国株（XOM, AMAT等）でも、抽出されたSECデータ（リスク、競争優位性、経営陣トーン等）がスコアに加味される。
- トークン上限エラーが回避され、安定して投資判断のプロンプトが最後まで生成・出力されるようになる。
